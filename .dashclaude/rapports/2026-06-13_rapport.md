### Itérations et corrections

1. **Placement** : Initialement prévu dans le drawer du pilotage, déplacé sur la page marque (plus logique car on veut voir toutes les saisons d'une marque).

2. **Tri chronologique** : ORDER BY par année extraite du nom de collection (DESC), puis AH avant PE dans la même année. Résultat : AH2026 en haut → PE2022 en bas.

3. **Taux de sortie v1** : Via ARTCOLART → valeurs aberrantes (3116% pour HOKA AH2023 car toutes les ventes all-time étaient comptées).

4. **Taux de sortie v2** : Via RECBRL/RECBR (articles effectivement reçus) → mieux mais toujours all-time.

5. **Taux de sortie v3** : Ventes scopées par période de saison (PE=jan-jul, AH=jul-jan+1) via CASE WHEN dans le SQL + REGEXP_SUBSTR pour extraire l'année du nom de collection. DERNIERE_VENTE reste non scopée pour le calcul de statut.

6. **Statut** : Initialement basé sur la dernière vente (<60j = "En cours"). Corrigé pour se baser sur la fin de période de la saison (`SEASON_TO` calculé depuis le nom de collection). PE2023 passait incorrectement "En cours" → maintenant "Terminé".

7. **Formule taux de sortie** : Changée de `qteVendue / qteRecue` à `(qteRecue - qteStock) / qteRecue` = taux total d'écoulement (tout ce qui est sorti, toutes périodes confondues).

8. **Deux colonnes ventes** : Ajout de `QTE_VENDUE_TOTAL` (all-time via SUM sans CASE) dans le SQL. Affichage en deux colonnes : "Vendu saison" (scopé) et "Vendu hors" (total - saison). Cache key bumpé à `v2` pour forcer le refresh.

## Architecture technique

### Requête SQL `PILOTAGE_ORDER_HISTORY`
- 6 paramètres (tous = marqueNom uppercased)
- Sous-requêtes : cmd (COMBCDE/COMBCDEL), cde-dates, rec (RECBR/RECBRL), ven (RECBRL + _ventes_daily avec CASE date), stk (RECBRL + AGRSTOCKCOUR), fou (RECBR + ARTFOURN avec ROW_NUMBER)
- Ventes : agrégation conditionnelle CASE WHEN pour scoper par période + SUM total
- SEASON_TO calculé dans le SELECT pour le statut côté client
- Tri : REGEXP_SUBSTR + CASE pour ordre chronologique

### API
- `GET /api/pilotage/order-history?marque=X`
- Pattern : `cached()`, `query<T>()`, `TTL.DEFAULT` (1h)
- Cache key : `pilotage-order-history:v2:${marque}`

### Frontend (page marque)
- Onglet lazy-loaded "Historique commandes"
- Statut calculé client-side via `seasonTo` vs `today`
- Taux de sortie = `(reçu - stock) / reçu × 100`, affiché uniquement si Terminé
- Badges colorés : bleu (En commande), ambre (En cours), vert (Terminé)
- Taux coloré : vert >=70%, ambre 40-69%, rouge <40%

### Lecture de bons de livraison par IA (nouvelle fonctionnalité)

Page `/receptions/bon-livraison` permettant d'uploader un bon de livraison PDF, de le faire analyser par Claude (API Anthropic), puis d'afficher un résumé structuré avec articles, photos et grilles de tailles.

**Fichiers créés :**
- `app/api/bon-livraison/parse/route.ts` — POST, envoie le PDF à Claude Sonnet pour extraction structurée
- `app/api/bon-livraison/init-table.ts` — Création auto de la table `bl_analyses` en MariaDB
- `app/api/bon-livraison/save/route.ts` — POST, sauvegarde une analyse en BDD
- `app/api/bon-livraison/list/route.ts` — GET, liste les analyses sauvegardées
- `app/api/bon-livraison/analyses/route.ts` — GET + DELETE, charger/supprimer une analyse
- `app/receptions/bon-livraison/page.tsx` — Page complète upload + affichage + vérification

**Fichiers modifiés :**
- `app/receptions/page.tsx` — Bouton "Lire un bon de livraison" ajouté dans le header

**Fonctionnalités :**

1. **Upload PDF** : Zone drag & drop, envoi à l'API Claude pour parsing
2. **Affichage structuré** : KPIs (nb BL, articles, pièces), header par BL (date, commande, fournisseur, marque), cartes articles avec photos Sport2000 CDN et grilles de tailles par couleur
3. **Photos** : URL pattern `https://media.sport2000.fr/photos_plateforme_digitale/{MARQUE}/app-{styleNo}-p.jpg` avec `&` → `AND` dans le nom de marque
4. **Tailles jeans** : Prompt Claude adapté pour combiner taille + longueur → format `29/32`, `30/30`, etc.
5. **Sauvegarde en BDD** : Table `bl_analyses` (id, file_name, supplier, brand, totaux, parsed_data JSON, created_at). Nom auto-généré : `{marque} — {date} — BL n°{numéros}`
6. **Historique** : Liste des analyses sauvegardées sous la zone d'upload, chargement en un clic, suppression avec confirmation
7. **Mode vérification** : Bouton "Vérifier réception", chaque cellule de taille devient un input. Vert = quantité OK, Rouge = écart. Barre de progression avec compteur OK/écarts
8. **Recherche articles** : Barre de recherche filtrant par n° style, nom, ou couleur pour retrouver rapidement un article en main

**Architecture :**
- Clé API Claude stockée dans `.dashclaude-data.json`
- PDF envoyé en base64 via content type `document` à l'API Anthropic
- Modèle : `claude-sonnet-4-20250514`
- Parsed data stockée en LONGTEXT (JSON) dans MariaDB
- Table créée automatiquement au premier appel (`CREATE TABLE IF NOT EXISTS`)
