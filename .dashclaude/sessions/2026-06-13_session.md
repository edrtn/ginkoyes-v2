## Décisions prises

1. **Page marque > Page pilotage** : L'historique multi-saisons n'a pas sa place dans le pilotage (qui est scopé à une saison). La page marque est le bon endroit.

2. **RECBRL/RECBR > ARTCOLART pour les ventes** : ARTCOLART associe trop d'articles à chaque collection (appartenance logique vs réception physique). Résultat : taux de sortie aberrants. RECBRL lie les articles aux collections via leurs bons de réception réels.

3. **Scoping des ventes par période** : Même via RECBRL, les ventes all-time gonflent le taux. Solution : CASE WHEN avec dates calculées depuis le nom de collection (PE=jan-jul, AH=jul-jan+1).

4. **Statut basé sur la fin de période** : La dernière vente all-time rendait des saisons anciennes "En cours" si l'article se vendait encore. Fix : `SEASON_TO` calculé depuis COL_NOM, comparé à today.

5. **Taux de sortie = (reçu - stock) / reçu** : Plus fiable que vendu/reçu car les ventes scopées ne captent qu'une partie des écoulements. Le stock restant dit la vérité.

6. **Deux colonnes ventes** : "Vendu saison" (scopé) + "Vendu hors" (total - saison) pour voir la répartition complète.

## Session 2 — Bon de livraison IA

### Objectif
Créer une page pour uploader un bon de livraison PDF, le faire lire par Claude API, et générer un résumé structuré avec articles, photos et grilles de tailles. Puis sauvegarder en BDD pour pouvoir y revenir.

### Fichiers créés
- `app/api/bon-livraison/parse/route.ts` — POST, parsing PDF via Claude API
- `app/api/bon-livraison/init-table.ts` — CREATE TABLE IF NOT EXISTS `bl_analyses`
- `app/api/bon-livraison/save/route.ts` — POST, sauvegarde analyse
- `app/api/bon-livraison/list/route.ts` — GET, liste analyses
- `app/api/bon-livraison/analyses/route.ts` — GET + DELETE, charger/supprimer
- `app/receptions/bon-livraison/page.tsx` — Page complète (upload, affichage, vérification, recherche)

### Fichiers modifiés
- `app/receptions/page.tsx` — Bouton "Lire un bon de livraison" dans le header

### Décisions prises

1. **Clé API dans `.dashclaude-data.json`** : Pas de variable d'env, lecture directe du fichier JSON côté serveur.

2. **PDF en base64 au lieu d'images** : Envoi du PDF complet via content type `document` (supporté par Claude). Pas besoin de convertir en images page par page.

3. **Modèle Claude Sonnet** (pas Opus) pour le parsing : Suffisant pour l'extraction structurée, plus rapide et moins cher.

4. **LONGTEXT au lieu de JSON** pour `parsed_data` : Compatibilité MariaDB plus large.

5. **Photos Sport2000 CDN** : URL pattern avec marque complète (`JACK AND JONES` au lieu de `JACK`), `&` remplacé par `AND`. Sans code couleur (pas disponible dans les BL fournisseurs).

6. **Tailles jeans combinées** : Prompt Claude adapté pour combiner taille colonne + Length → `29/32`. Chaque longueur = entrée couleur séparée `"Blue Denim (L32)"`.

7. **Nom de sauvegarde auto** : `{marque} — {date} — BL n°{numéros}` généré depuis les données parsées, pas depuis le nom du fichier PDF.

8. **Mode vérification** : Inputs dans la grille de tailles, comparaison attendu vs reçu avec feedback couleur (vert/rouge).

9. **Recherche articles** : Filtre en temps réel par n° style, nom ou couleur pour retrouver un article quand on l'a en main.

### Commandes exécutées
- `npx tsc --noEmit` — Vérification TypeScript à chaque étape
- `mkdir -p app/api/bon-livraison/{parse,save,list,analyses}`
- `git add + git commit` — 47 fichiers, +6450 lignes

### Points d'attention
- La table `bl_analyses` est créée automatiquement au premier appel API (flag `initialized` en mémoire)
- Les checks de vérification sont en mémoire côté client (pas sauvegardés en BDD pour l'instant)
- Les photos sans code couleur utilisent le pattern `app-{styleNo}-p.jpg` — certaines peuvent ne pas exister (placeholder affiché)
